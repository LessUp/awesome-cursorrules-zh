# 学院路径

“学院路径”不是教程集合，而是把不同读者组织成**可递进的学习与采用链路**。

## 三条主路径

### 1. 价值判断路径

适合技术负责人、面试官和开源项目评审者。

1. 读 [项目使命](../introduction/mission)
2. 看 [系统总览](../architecture/system-overview)
3. 用 [研究引用](../research/references) 和 [规则索引](../rules/) 交叉验证

### 2. 架构判断路径

适合需要理解主题层、内容系统和发布层关系的架构师。

1. 看 [系统总览](../architecture/system-overview)
2. 补读 [站点蓝图](../architecture/blueprint)
3. 用 [规则组合模式](../rules/composition-patterns) 验证证据层是否服务于总叙事

### 3. 落地与治理路径

适合维护者、贡献者和团队内部 DevEx 负责人。

1. 读 [维护者课程](./maintainer-curriculum)
2. 按 [团队接入](../guides/team-onboarding) 校对流程
3. 回到 [规则索引](../rules/) 选择当前阶段需要启用的规则组合

## 为什么叫“学院”

因为项目已经不是单页说明书。它必须同时承担：

- **导读**：帮助读者建立判断
- **课程**：帮助不同角色形成阅读顺序
- **证据**：帮助团队把规则转为工程实践

这就是“学院、指南、导读”三层逻辑的核心。
