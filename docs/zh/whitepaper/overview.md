# 项目总览

## 为什么要把站点重构成白皮书

当前项目真正的核心并不是页面本身，而是 `rules/` 目录里持续增长的 Cursor 规则资产。  
如果 GitHub Pages 只是把这些规则目录重新列一次，它的价值很快就会见顶：用户看到了“很多规则”，却不知道：

1. 为什么要使用规则而不是临时提示词。
2. 如何判断哪些规则适合当前项目。
3. 团队应该怎样把规则纳入治理和版本管理。
4. 遇到规则冲突、失效或演进需求时该如何处理。

因此站点需要承担新的角色：**技术白皮书**、**架构展示站**、**操作方法论入口**。

## 站点的三个核心主张

### 1. 规则是知识资产，不是附件

`.cursorrules` 不是写给 AI 的装饰性配置，而是组织开发经验、架构边界和协作纪律的承载体。  
一条高质量规则，本质上等价于一段被结构化表达的工程知识。

### 2. 好规则必须有上下文

单独看某个规则文件，往往无法理解它的适用场景、与其他规则的组合关系、以及它在团队中的维护方式。  
所以站点要负责补齐这些上下文：适用对象、采用路径、冲突处理、升级策略、参考资源。

### 3. 文档站应当解释系统，而不是仅仅展示文件

重构后的 GitHub Pages 会把项目拆成四个互相配合的内容域：

- **白皮书**：解释定位、原则和采用模型
- **架构**：解释内容系统和发布结构
- **方法论**：解释如何落地、治理、迭代
- **资源**：连接上游、官方和延伸阅读

## 核心判断标准

一个合格的新站点，不是“更漂亮”而已，而是应该让用户能完成以下动作：

1. 在 3-5 分钟内理解这个项目的定位。
2. 在 5-10 分钟内找到适合自己的规则起点。
3. 在 10-20 分钟内理解如何把规则纳入团队协作。
4. 在需要深入研究时，能继续沿着资源网络往外扩展。

## 设计原则

### 轻框架，重结构

视觉上借鉴 kimi-cli 文档站的干净框架：清晰导航、轻量首页、低摩擦信息密度。  
但内容上不做“产品说明页”的重复，而是更强调结构、关系和实践指导。

### 内容域优先于页面堆砌

新增页面不是为了填充导航，而是围绕明确的认知任务组织：理解项目、理解结构、采取行动、继续探索。

### 以长期收益为优先

允许破坏旧的信息结构，只保留真正重要的资产和路径。  
如果某个旧页面无法服务新的定位，就应该被边缘化或重组。
