# 白皮书总览

本页把首页的 executive brief 展开成完整叙事：先解释为什么值得采用，再说明系统如何组织，最后给出可执行的启动方式。

## 先回答为什么值得采用

这个项目的长期价值，不在于“收集了多少 `.cursorrules`”，而在于它把分散的工程经验升级成组织级工程知识系统。

### 组织资产

规则可以承载团队关于命名、目录、接口边界、测试要求和评审习惯的共识。与一次性提示词不同，规则进入仓库之后会留下版本历史、审阅记录和迭代轨迹。

### 治理成本

当团队把常见要求写回规则，沟通就不再依赖口头提醒。AI 协作和人工协作共享同一份知识边界，治理成本会随着资产复用而下降。

### 复用收益

规则资产一旦成形，就可以在新仓库、新团队和新技术域之间复用。首页因此必须先说明价值逻辑，而不是先展示目录。

## 再回答系统如何组织

项目被组织成“价值叙事 → 系统结构 → 操作路径 → 证据层”的固定阅读顺序。

| 层级 | 主要目录 | 作用 |
| --- | --- | --- |
| 白皮书层 | `docs/zh/whitepaper/*` | 定义价值主张、采用判断和阅读顺序 |
| 架构层 | `docs/zh/architecture/*` | 解释 `rules/`、主题层和发布链路如何协同 |
| 操作层 | `docs/zh/playbook/*` | 说明角色入口、采用节奏和最小治理动作 |
| 证据层 | `rules/` 与 `docs/zh/rules/*` | 提供规则样本、分类导航和具体证据 |

这套结构让首页不再是入口拼盘，而是一个按决策顺序推进的白皮书展厅。

## 最后回答怎样开始

建议按照“判断价值 → 确认结构 → 选择路径 → 查看证据”的顺序推进：

1. 先读 [决策者摘要](./decision-brief)，确认它是否符合你的组织目标。
2. 再读 [架构蓝图](../architecture/blueprint) 与 [信息图谱](../architecture/information-graph)，理解系统边界。
3. 然后进入 [采用路径](../playbook/adoption-path) 与 [角色路径](../playbook/role-paths)，选择你的行动方案。
4. 最后再进入 [规则证据库](../rules/)，把需要的规则样本带回仓库。
