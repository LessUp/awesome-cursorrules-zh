# 决策者摘要

本页面向技术负责人与架构师，用最短路径回答三个问题：为什么值得采用、系统如何组织、怎样开始。

## 先回答为什么值得采用

如果这个项目继续被呈现成“规则列表站”，管理层只能看到数量，无法看到组织资产沉淀、治理成本下降和复用收益上升。真正值得投资的不是 132+ 个文件本身，而是把工程经验写成可审阅、可复用、可迭代的知识资产。

- **组织资产**：规则把命名、目录、测试、架构边界固化成团队共享语言。
- **治理成本**：规则进入仓库和评审流程后，团队不需要在每次协作中重复口头对齐。
- **复用收益**：同一套规则组合可以被不同仓库复用，持续放大历史经验的价值。

## 再回答系统如何组织

站点被重构为一条清晰的解释链路，而不是首页直冲规则列表。

1. **`rules/` 是资产源**：保存规则正文与说明，承担证据库职责。
2. **白皮书与操作页是叙事层**：解释为什么采用、如何判断、怎样落地。
3. **架构图谱是系统层**：说明内容如何穿过主题、导航和 GitHub Pages 被发布。
4. **资源网络是外延层**：把官方文档、上游仓库与参考资料连接回来。

这样的组织方式让架构师先理解系统边界，再决定如何把规则接入团队工程流。
