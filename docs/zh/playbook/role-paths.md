# 角色路径

本页为技术负责人、架构师、平台负责人和贡献者提供不同的进入顺序，确保每个角色都沿着同一套叙事推进，而不是各自只看局部页面。

## 先回答为什么值得采用

角色路径的意义，在于让不同职责的人看到同一套系统的不同切面：

- **技术负责人** 需要确认投资回报、治理收益与组织影响。
- **架构师** 需要确认结构边界、发布链路与内容责任分层。
- **平台负责人** 需要确认维护模型、扩展方式与推广节奏。
- **贡献者** 需要知道自己新增内容时会落在哪一层，而不是把首页重新拉回目录思维。

每个角色都从价值或结构开始，只有在认知路径完成之后，才进入规则证据库查看样本与细节。

## 最后回答怎样开始

你可以直接按角色执行下面的第一步动作：

1. **技术负责人**：把首页和决策者摘要当作 15 分钟 briefing，先判断是否值得推动试点。
2. **架构师**：拿站点蓝图和信息图谱梳理 `rules/`、文档与发布面之间的关系。
3. **平台负责人**：根据采用路径设计试点节奏和治理回路。
4. **贡献者**：先理解系统叙事，再向规则证据库补充高质量样本。
