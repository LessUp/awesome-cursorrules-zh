# 运行模型

## 为什么需要运行模型

规则一旦开始服务团队，就不再只是个人偏好。  
它会影响代码评审、目录结构、测试习惯和交付质量，因此必须有基本治理。

## 最小可行运行模型

### 角色

| 角色 | 责任 |
| --- | --- |
| 规则维护者 | 维护模板、清理冲突、组织更新 |
| 项目负责人 | 决定哪些规则进入项目 |
| 贡献者 | 提出问题、补充经验、提交改进 |

### 流程

1. 发现问题：AI 输出不稳定、规则过时、上下文冲突
2. 提出修改：更新规则文件或文档说明
3. 评审验证：至少跑一次真实开发任务验证
4. 合并沉淀：记录在 changelog 或团队文档中

## 推荐治理动作

### 规则评审

- 新增规则时说明适用场景
- 修改规则时说明为什么变
- 删除规则时说明替代路径

### 版本回顾

每次 Cursor 或主框架升级后，检查：

- 是否有过时约束
- 是否需要加入新的工具链规则
- 是否有新冲突需要拆分

### 模板库建设

对频繁复用的组合，推荐建立内部模板：

- 前端默认模板
- 后端默认模板
- 全栈模板
- 安全敏感模板

## 站点在运行模型中的作用

这个 GitHub Pages 站点不只是对外展示，也是运行模型的一部分：

- 白皮书负责统一认知
- 架构页负责解释边界
- 方法论负责降低落地摩擦
- 资源页负责提供外部参照

如果未来要建设更强的组织级规则平台，这套站点结构可以继续扩展，而不必重来。
