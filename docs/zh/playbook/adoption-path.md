# 采用路径

## 推荐起步顺序

### 第一步：用业务上下文划定边界

先回答三个问题：

1. 项目是单技术栈还是多技术栈？
2. 团队更需要“提效”还是“统一”？
3. 你现在最痛的点是生成质量、风格漂移，还是架构失控？

## 第二步：选择规则组合

| 场景 | 推荐起点 |
| --- | --- |
| 个人 Web 项目 | `general` + 主框架规则 |
| 团队全栈项目 | 根目录通用规则 + 前后端子目录规则 |
| 安全或基础设施敏感项目 | 基础规则 + `security` / `devops` 领域规则 |
| AI / 数据团队 | 基础规则 + `ai` / `data-science` / `data` |

## 第三步：先验证，再扩张

不要一开始就合并大量规则。  
推荐先跑一个真实开发回合，观察三个信号：

- AI 是否开始遵守命名与目录约束
- 生成代码是否更接近团队风格
- 是否出现规则冲突或上下文噪声

## 第四步：把经验写回规则

如果某些补充提示词你总在重复说，那它们很可能应该成为规则的一部分。  
常见候选包括：

- API 响应格式
- 测试要求
- 目录结构
- 错误处理策略
- 架构边界

## 第五步：建立轻量治理

推荐的最小治理动作：

1. `.cursorrules` 纳入 Git
2. 规则变更进入 PR
3. 每次重大版本升级后回看规则是否失效
