# 内容系统

## 内容系统的核心问题

对于这类项目，最大的风险不是“没有内容”，而是内容很多却没有结构。  
一旦站点只按文件或目录罗列，用户就很难理解：

- 先看什么
- 为什么看
- 看完之后下一步是什么

因此内容系统必须解决“认知流”的问题。

## 新的内容域划分

### 白皮书（Whitepaper）

回答：这个项目为什么值得存在。  
适合放：定位、原则、采用模型、判断标准。

### 架构（Architecture）

回答：这个站点和这套知识资产是如何组织的。  
适合放：信息架构、目录职责、发布结构、设计原则。

### 方法论（Playbook）

回答：你在真实项目中应该如何行动。  
适合放：采用路径、治理方法、模板、故障排除。

### 资源（Resources）

回答：如果还要继续深入，应该去哪。  
适合放：官方文档、上游仓库、相关工具、延伸阅读。

## 中英文双站策略

| 内容类型 | 中文 | 英文 |
| --- | --- | --- |
| 核心定位与方法论 | 优先完整表达 | 保持同步与可读 |
| 规则目录与分类 | 中英一致 | 中英一致 |
| 资源链接 | 链接共享 | 链接共享 |

原则是：**对外文档优先中文，但结构保持双语一致**。

## 侧边栏设计原则

1. 按主题域分组，而不是按“某个时期写了哪些页面”分组。
2. 每个主题域控制在 2-6 个核心入口，避免侧边栏膨胀。
3. 把旧页面吸收到方法论或资源域中，而不是继续单独漂浮。

## 搜索策略

本次设计延续本地搜索，但搜索只是补充，不应成为用户唯一入口。  
理想状态下，用户即使不搜索，也能从首页和导航快速找到第一跳。

## 内容增长方式

后续新增内容时，推荐遵循下面的判断：

```text
解释“为什么” -> 白皮书
解释“怎么组织” -> 架构
解释“怎么做” -> 方法论
解释“去哪里看更多” -> 资源
```

这会让内容系统保持可预测性，也更适合长期维护。
