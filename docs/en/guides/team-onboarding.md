# Team Onboarding

This page turns adoption into an execution checklist.

## 1. Decide the onboarding goal

Before you adopt the project, answer three questions:

1. Are you building a shared prompting baseline or a broader engineering knowledge layer?
2. Do you need a fast pilot or a maintainable long-term system?
3. Do you mainly need rule samples, or do you need a public-facing project shell that demonstrates maturity?

## 2. Pick the right entry surface

- **Pilot evaluation**: start with the [Project Mission](../introduction/mission) and [Composition Patterns](../rules/composition-patterns)
- **Team rollout**: start with the [Academy Path](../academy/learning-path) and [Best Practices](../best-practices)
- **Repository maintenance**: start with the [Maintainer Curriculum](../academy/maintainer-curriculum) and [Rule Template Guide](./rule-template)

## 3. Define a minimal baseline

Use a three-part starting set:

- one general rule
- one primary-stack rule
- one governance rule from security, data, or platform

That is enough to create a team baseline without front-loading too much complexity.
