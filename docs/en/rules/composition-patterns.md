# Composition Patterns

This page helps you decide which rule combination to validate before browsing category pages.

## Return to the executive context

- [Decision Brief](../whitepaper/decision-brief)
- [Site Blueprint](../architecture/blueprint)
- [Adoption Path](../playbook/adoption-path)

## Base pattern

- General + primary stack
- Start with [General](/en/rules/general) and the page for your main stack.

## Expansion pattern

- Primary stack + security / data / platform
- Add [Security](/en/rules/security), [Data](/en/rules/data), or [Platform](/en/rules/platform) when delivery risk grows.

## Upgrade signals

- Repeated instructions should become project-specific rules.
- Repeated review boundaries are a signal to formalize a team baseline.
- Governance belongs in the shell and playbook, while rule pages remain the evidence layer.
