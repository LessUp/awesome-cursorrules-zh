# GitHub Pages 白皮书级重构设计稿

## 1. 背景与目标

当前站点已经完成一轮从“规则目录”到“白皮书型文档站”的升级，但还存在三个核心短板：

1. **信息主轴不够锋利**：`academy / guides / research / rules` 的结构比旧版更成熟，但仍未完全围绕“系统架构设计、算法介绍、技术白皮书与性能、参考资料”四条主轴收敛。
2. **视觉系统尚未形成压迫性优势**：现有主题有良好的 token 基础和若干自定义组件，但首页与专题页的叙事张力、图形语义、专业感层次仍不够统一，距离“技术白皮书 + 极客美学”的标杆还有明显差距。
3. **图示与生成数据层仍偏轻量**：目前 `siteFacts` 只生成计数级数据，无法支撑更高级的系统图、算法页、性能页和证据链可视化，也限制了页面从“说明”升级为“可审阅的技术陈述”。

本次重构的目标不是修补，而是以长期收益为优先，重写 GitHub Pages 的**信息架构、可视化语言、数据生成层与核心叙事页面**，把站点提升为面向高级开发者与开源审稿人的“研究级技术门户”。

## 2. 设计边界与假设

### 已知边界

- 保持技术栈在 **Node.js + VitePress + 自定义主题组件** 范围内演进，不引入额外前端框架。
- 中文文档仍然是主要叙事语言，英文页面保持同步但可略微收敛表达密度。
- `rules/` 依然是全站唯一权威资产源，站点上的统计、分层、路径与证据陈述应尽量从资产源派生。

### 本次自主假设

由于当前为自动执行模式，本设计稿直接采用以下假设推进，不等待额外确认：

- **信息架构将被激进调整**，允许重命名导航、侧边栏分组和首页叙事顺序。
- **双语页面同步升级**，但中文页面承担更强的“白皮书正文”角色，英文页面保持同构但更凝练。
- **算法介绍** 不解释模型内部算法，而解释本项目如何进行“规则资产检索、组合、决策、验证与发布”的工程算法与策略。
- **参考资料** 将包含论文、方法论文章、竞品仓库、相关标准与演进思考，而不是简单友情链接。

## 3. 备选方案

### 方案 A：仅强化首页与主题样式

在保留现有 IA 的基础上重做首页、增强组件视觉表现、修复 SVG 深浅色适配问题。

**优点**：改动小，风险低，交付快。  
**缺点**：无法从根本上解决站点叙事主线分散、数据层太薄、技术深度不够的问题。

### 方案 B：重写信息架构与首页叙事，局部升级数据层

围绕“导读 / 系统架构 / 算法机制 / 性能白皮书 / 参考资料 / 规则证据”重组 IA，并扩展生成数据用于新的图示与指标模块。

**优点**：能显著提升专业度与结构完整性。  
**缺点**：如果只做到局部，视觉语言与内容深度仍可能出现“新旧混搭”的违和感。

### 方案 C：全面重构为研究级技术白皮书站（推荐）

同时重构 **IA、首页、专题页、数据生成层、图示组件、主题 token、引用系统与验证基线**。首页变成“总论”，专题页分别承担系统架构、算法、性能、参考资料等角色；所有图形和关键指标尽量建立在生成数据与统一设计 token 之上。

**优点**：最符合本次“颠覆式重构”的目标，长期收益最高，站点辨识度与可信度都会显著提升。  
**代价**：涉及多文件、多页面、多组件与测试更新，但这是本次任务明确追求的方向。

## 4. 最终设计蓝图

### 4.1 核心定位

站点定位从“规则合集文档站”升级为：

> **Cursor Rules 的系统设计白皮书、规则编排算法说明书、性能与证据陈述平台，以及面向高级开发者的研究型参考门户。**

首页不再只承担导航入口，而是承担“总论”职责：用最短路径回答审稿人最关心的四个问题：

1. 这个项目解决什么认知问题？
2. 这个仓库背后是否有清晰的系统架构？
3. 它如何做规则检索、组合、验证与演化？
4. 有哪些客观证据、性能指标与参考文献支撑其价值？

### 4.2 信息架构

新的主导航建议收敛为六条主轴：

1. **导读**：定位、读者地图、快速评审路径
2. **系统架构**：站点结构、资产流水线、发布层、内容系统
3. **算法机制**：规则检索、组合策略、决策树、质量校验
4. **性能白皮书**：体量、覆盖、构建链路、可维护性指标、体验优化
5. **参考与演进**：参考文献、竞品开源项目、演进笔记、设计决策
6. **规则证据库**：规则索引、领域分布、组合模式、关键示例

现有 `academy` 和 `guides` 不再作为顶级叙事中心，而是被吸收为：

- 导读下的评审路径 / 学习路径
- 规则证据库下的实践引导
- 参考与演进下的维护与贡献入口

这样可以让站点更接近论文式结构：**先立论，再解释系统，再展示方法，再陈述证据，最后附参考资料与索引。**

### 4.3 数据生成层

现有 `docs/scripts/sync-site-facts.mjs` 只生成规则总数和头部类别，无法支撑高级叙事。本次将它升级为**站点知识快照生成器**，输出以下结构：

- `ruleCount`、`domainCount`、`localeCount`
- `topDomains`
- `categoryDistribution`
- `taxonomyDepth`
- `coverageBuckets`
- `curationSignals`（如 README 覆盖、规则密度、分布均衡性）
- `qualitySignals`（如最活跃类别、组合模式提示、知识图谱节点数）
- `timelineSignals`（静态推导型演进节点，用于首页与白皮书页）

生成数据不追求“伪实时复杂分析”，而追求**足够可信、可复用、能稳定驱动 UI 与文档论点**。

### 4.4 视觉语言

视觉设计采用“**学术仪表板 + 极客档案馆 + 开源研究论文**”的混合语言：

- 基础层：保留 OKLCH token 体系，但改进对比度、边框、发光与图形色阶
- 页面层：通过更强的 section framing、grid overlays、evidence cards、lab-style captions 提升白皮书质感
- 图形层：所有核心图示统一使用一套 figure token，不允许散落硬编码颜色
- 文本层：标题更具展示感，正文保持克制，提高高级感而不是堆砌动画

深浅色模式的策略不是简单调亮/调暗，而是：

1. 为图示定义独立 token
2. 保证文本、连线、背景层、强调层都有明确的 light/dark 映射
3. 让 SVG 组件尽量使用 CSS 变量和 `currentColor`
4. 避免依赖 `filter: invert()` 这类后处理作为主要方案

### 4.5 图示系统

图示要从“漂亮的块”升级为“有解释力的图”。

本次会重建以下可视化组件：

- **系统拓扑图**：展示资产层、生成层、叙事层、发布层、证据层之间的依赖
- **规则编排流水线图**：展示从“规则发现 → 分类 → 组合 → 证据化 → 发布”的流程
- **性能与质量面板**：以研究仪表板风格呈现覆盖广度、内容密度、构建路径与维护信号
- **引用矩阵 / 竞品对照卡**：展示参考文献与开源项目对本站不同模块的影响

所有图示都必须：

- 在浅色与深色模式下可读
- 支持 Markdown 页面复用
- 不依赖位图
- 尽量由统一组件驱动，而不是每页散写 SVG

### 4.6 内容重构策略

内容层按“主张 → 结构 → 方法 → 证据 → 参考”展开：

- **导读**：强化项目使命、受众、评审路径
- **系统架构**：解释站点为什么这样组织、生成数据如何服务内容、主题层如何支持叙事
- **算法机制**：解释规则检索、组合、决策树、升级路径和质量校验逻辑
- **性能白皮书**：解释为什么这个仓库在组织质量、知识密度、导航效率、构建稳定性上更优秀
- **参考与演进**：为所有重要设计选择提供出处、竞品观察与后续演化思考
- **规则证据库**：把规则页重新定义为“证据附件”，而不是站点唯一主角

### 4.7 工程实现结构

实现上分为四层：

1. **Data Layer**  
   升级 `sync-site-facts.mjs` 与生成数据结构，为页面和图形组件提供更丰富的快照。

2. **IA Layer**  
   重写 `atlas.ts` 的主导航与侧边栏分组，让信息架构直接反映白皮书逻辑。

3. **Presentation Layer**  
   新增一组研究感组件，用于 hero、diagram、metrics、dossier、reference matrix、pipeline。

4. **Content Layer**  
   重写首页与关键专题页；补足算法、性能、参考资料、演进思考等高价值页面。

### 4.8 错误处理与兼容策略

这次重构不以向后兼容旧路径为目标，但仍保持以下工程安全性：

- 所有生成数据在缺失字段时以显式空状态渲染，而不是静默失败
- 页面组件依赖统一数据接口，避免每页各自做字符串拼接
- 测试基线会更新为新的 IA 与新组件，确保结构性回归能被尽早发现

### 4.9 验证策略

验证不只关注“能否构建”，还关注“结构是否真的升级”：

- 更新 `tests/site-ia.test.mjs` 断言新的导航、页面与组件注册
- 保持 `npm run check` 为最终质量门禁
- 继续保留并强化 SVG 主题适配要求，杜绝浅深色图示失效

## 5. 页面蓝图

### 首页

首页将升级为“项目总论”，分为：

1. Thesis Hero
2. Architecture Radar
3. Rule Orchestration Pipeline
4. Performance & Coverage Ledger
5. Evidence Atlas
6. References & Evolution Dossier

### 系统架构页

从“系统总览”升级为更像 ADR + 研究说明的页面，重点解释：

- 权威资产源
- 生成数据层
- 叙事与信息架构
- 可视化与主题机制
- 发布链路与 GitHub Pages 输出

### 算法机制页

新增或重写算法专题，解释：

- 规则检索策略
- 分类聚合逻辑
- 组合模式
- 决策树与升级路径
- 内容验证与质量闸门

### 性能白皮书页

将性能从“构建快慢”扩展为：

- 内容覆盖密度
- 导航信息效率
- 生成链路稳定性
- 维护复杂度控制
- 图示与主题适配可靠性

### 参考与演进页

构建真正的 scholarly references 模块，增加：

- 参考文献分组
- 相关开源项目对照
- 设计影响映射
- 演进思考与下一阶段研究问题

## 6. 预期结果

重构完成后，站点应具备以下效果：

- 第一屏就能体现“这不是普通导航站，而是一个有方法论、有结构解释、有证据链的技术作品”
- 高级开发者可以沿着清晰逻辑快速完成评审
- 视觉风格在浅色/深色模式下都具有一致的专业感
- 图示不再是装饰，而是承担结构说明职责
- 文档内容具备白皮书、架构说明与参考文献体系的完整感

## 7. 实施建议

实施上建议采用以下顺序：

1. 先升级生成数据与 IA
2. 再重建主题组件与 token
3. 然后重写首页与核心专题页
4. 最后更新双语内容、测试与校验

这能保证每一层都有明确的依赖边界，并避免“先写大量内容、后面又因结构变化重做”的浪费。
